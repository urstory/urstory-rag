# RAG 품질 테스트 보고서 — 공개 테스트 데이터셋 (2026-03-07)

## 개요

공개 테스트 데이터셋(`test_files/public_dataset/`)을 사용한 첫 번째 품질 테스트 결과이다.
기존 비공개 문서(사내 PDF/MD) 대신 **저작권 없는 공공 문서**로 구성된 데이터셋으로,
누구나 동일한 조건에서 RAG 품질을 재현할 수 있도록 설계되었다.

| 문항 | 점수 | 원인 |

| Q55 (혼합경제 정의) | 10 | bok_easy_economics.pdf에서 관련 청크 검색 실패 |
| Q56 (보이지 않는 손) | 0 | bok_easy_economics.pdf에서 관련 청크 검색 실패 |
| Q57 (GDP 정의) | 0 | bok_easy_economics.pdf에서 관련 청크 검색 실패 |

3건 모두 `bok_easy_economics.pdf`에서 발생했다. 이 문서는 335개 청크로 분할되었으나,
경제 개념 정의를 묻는 질문에 대해 적절한 청크가 검색되지 않았다.

원인 분석:
- 해당 PDF는 일러스트/도표가 많은 교육용 자료로, 텍스트 추출 시 레이아웃 정보가 손실된다.
- 개념 정의가 여러 페이지에 걸쳐 시각적으로 표현되어 있어, 단일 청크에 정의 전체가 포함되지 않는다.
- 키워드 검색(Elasticsearch)에서도 "혼합경제", "보이지 않는 손" 같은 핵심어가 추출된 텍스트에 온전히 포함되지 않았다.

### ERROR 문항 분석 (5건)

| 문항 | 원인 |
|------|------|
| Q33, Q52~Q54, Q67 | Cross-Encoder 리랭커 메모리 폭발 (128GB 버퍼 요청) |

테스트 도중 2,200개 이상의 청크가 리랭커에 입력되면서 메모리 한도를 초과했다.
테스트 후 다음 두 가지 수정을 적용하여 해결했다:

1. `hybrid.py`: 리랭커 입력 후보 수를 `reranker_top_k * 4` (=32)로 제한
2. `korean.py`: Cross-Encoder 입력 텍스트를 512자로 절단

수정 후 Q67 재테스트에서 정상 동작을 확인했으며, 나머지 4건도 동일 원인이므로 재실행 시 정상 처리될 것으로 예상한다.

청크 수가 약 2.8배 증가했음에도 불구하고 성능 하락은 제한적이다.
점수 차이의 주된 원인은 PDF 텍스트 추출 품질이며, RAG 파이프라인 자체의 성능 저하는 아니다.

## 성능 평가 의견

### 강점

1. **Markdown 문서 처리 능력이 탁월하다.** 32문제 중 31문제에서 만점을 기록했다. 구조화된 텍스트에 대한 검색-생성 파이프라인이 안정적으로 동작한다.

2. **대량 청크 환경에서도 검색 정확도를 유지한다.** 2,239개 청크에서도 하이브리드 검색 + 리랭커 조합이 관련 문서를 정확히 찾아냈다. PDF 문서 질문(ERROR/FAIL 제외)에서도 평균 80점 이상을 달성했다.

3. **가드레일이 적절히 동작한다.** 문서에 정보가 없는 경우(Q40, Q42 등) "정보를 찾을 수 없다"고 정직하게 응답하여 할루시네이션을 방지했다. Judge도 이를 100점으로 평가했다.

### 개선 필요 사항

1. **PDF 텍스트 추출 품질 개선이 필요하다.** 일러스트/도표 중심 PDF(bok_easy_economics.pdf)에서 개념 정의가 온전히 추출되지 않는다. OCR 기반 추출이나 레이아웃 인식 파서 도입을 검토할 필요가 있다.

2. **Contextual Chunking의 청크 크기 제어가 필요하다.** 문맥 정보를 청크 앞에 붙이면서 일부 청크가 임베딩 모델의 토큰 한도(8,192)를 초과했다. 현재 tiktoken 기반 절단으로 대응 중이나, 청킹 단계에서 크기를 사전 제어하는 것이 바람직하다.

3. **경제 용어 사전형 문서에 대한 검색 전략이 필요하다.** `bok_economic_terms_700.pdf`(1,050 청크)처럼 용어 정의가 나열된 문서에서는 기존 하이브리드 검색이 잘 동작하지만, 교육용 서술형 PDF에서는 추가적인 쿼리 확장이나 청킹 전략 조정이 필요하다.

## 테스트 재현 방법

```bash
# 1. 공개 데이터셋 문서를 관리자 UI에서 업로드 (test_files/public_dataset/)
# 2. 인덱싱 완료 대기
# 3. 품질 테스트 실행
cd backend
.venv/bin/python ../test_files/run_quality_test.py
```

결과는 `test_files/quality_test_results.json`에 저장된다.
